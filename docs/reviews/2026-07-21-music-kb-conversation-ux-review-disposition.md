# Music KB 对话 UX review disposition

日期：2026-07-21

状态：已记录 review 结论；本文件不引入新的用户行为规则

范围：`plugins/music-kb/skills/music-kb/SKILL.md` 的用户端检索和对话流程，以及为这套流程编写的 conversation-UX metric pack。

不在范围内：周更、下载、CNB、数据库发布、peer 同步、后台脚本复杂度、整包插件的 marketplace 元数据，以及提示词生成功能本身。

## 一句话结论

当前用户端 UX 没有需要立即阻塞发布的 review 问题；下一步应把已确认原则作为回归基线，并补一层真实对话 benchmark，而不是继续凭静态规则改写 Skill。

## 证据边界

### 用户端 Skill（本次 review 的目标）

- `plugin-eval analyze` 目标为 `plugins/music-kb/skills/music-kb`：A / 100，0 个失败、0 个警告；唯一信息项是没有覆盖率产物。
- `conversation-ux` custom metric pack：8/8 checks 通过，coverage 100%，failed checks 0。
- 当前 Skill 的静态检查覆盖：分支上限和排序、分支选择后的上下文延续、最小修复问题、渐进式结果展示、追加 / 换批方向和展示语义、用户可见的后续动作指引、证据和 `listen_url` 边界、未决语义不得被写成固定默认值，以及自然语言入口示例。
- 真实校准请求和用户确认记录在 [头脑风暴记录](../brainstorms/2026-07-20-music-kb-conversation-ux.md) 的 Round 17；它证明了“温暖”需要至少区分情绪方向和音色方向，但不把这三个方向写成通用分类表。

### 整个插件包（不作为本次 UX 结论）

对 `plugins/music-kb` 根目录运行整包评估得到 26/100，主要提示：

- `plugin.json` 缺少 `websiteURL`、`privacyPolicyURL`、`termsOfServiceURL`；
- 包含所有技能时的静态 deferred/invoke token 估算偏高；
- 后台 Python 文件存在复杂度和长行提示。

这些结果来自插件级目标，不是用户端 retrieval Skill 的行为证据。它们应另开维护任务，不能被写成当前 UX 改动的阻塞项。

| R2 | 主观宽请求可能有多个合理方向 | 已解决 | 保留最多三个有证据的完整分支、最可能方向优先和跨分支重叠标注。 |
| R3 | 大结果集可能造成结果墙 | 已解决但需观察 | 保留方案 1+：小结果集全量，大结果集代表性首轮并可继续展开；不锁定首轮具体数字。 |
| R4 | “再来一些”与“换一批”是否保留方向及如何展示 | 已解决 | 两者都保留当前方向；前者追加新结果，后者替换当前展示，旧结果展示语义已确定。 |
| R5 | 模糊多标签的交集/并集未定 | 延后 | 不声明通用默认语义；只有当前请求足够明确时才按该请求执行。 |

| R8 | 整包评估的 manifest/token/后台代码提示 | 范围外 | 不在本次用户端 UX 任务中修复，另行建立维护事项时再处理。 |
| R9 | 用户知道 Skill 规则，却不知道如何在回复中触发“再来一些”或“换一批” | 已解决 | 在可继续展开的回复中紧跟一段白话指引；用户实际调用时可用一句话确认追加或替换，不在无关回复重复教程。 |
| R10 | 当前方向结果不足时，既要允许放宽条件，也要保留相邻方向 | 已确定边界，呈现方式待定 | 不静默放宽或切换；先告知结果不足，再把两条路径交给用户选择。下一轮决定先后、并列或轻量取样。 |
| R11 | 结果不足时如何把两条路径呈现给用户 | 已解决 | 采用并列呈现：一个可放宽条件 + 一至两个相邻方向；用户选择前不检索、不放宽、不自动切换。候选生成边界和数量仍待定。 |
| R12 | 并列选项是否采用固定数量 | 已确定方向，护栏待定 | 采用完全动态数量；只展示有证据且有区分度的路径，不为填满数量制造选项。下一轮决定动态策略的停止护栏。 |
| R13 | 动态数量是否默认依赖用户再请求展开 | 已解决 | 不把“展开”设为必要步骤；优先压缩单项表达，并在一次回答中覆盖所有重要、不同且可扫描的路径。重复、弱证据或明显破坏可读性的内容才延后。 |
| R14 | 两个候选方向何时值得分别展示 | 已解决 | 必须同时具备可验证的检索证据差异和会改变用户选择的实际价值差异；只有一项满足时不拆成独立方向。 |
| R15 | 结果不足时优先建议放宽哪个条件 | 已解决 | 用户意图优先：建议放宽最不核心的条件，不以最大结果增量覆盖用户已强调、选定或纠正后保留的条件；核心程度按 R16 的完整对话规则判断。 |
| R16 | 如何判断条件的核心程度 | 已解决 | 使用完整对话：当前选定方向和纠正后保留的条件优先，其次是明确强调、重复表达和原始措辞；仍无法区分时只问一个最小问题，回答前不改变条件。 |
| R17 | 完整对话仍无法区分时怎样提出最小问题 | 已解决 | 中性并列两个可放宽条件，各用一句话说明放宽什么、保留什么及用户可感知的影响；用户选择前不检索，选择后不再重复确认。 |

## 真实对话 benchmark（计划，不在本轮执行）

### 目标

验证用户是否能更快得到可用候选、看懂 Skill 采用的理解、低成本纠正方向，并且不会因为库变大而被结果淹没。

### 场景集

1. 清晰的中文标签或别名请求。
2. “有氛围感”这类宽范围主观请求。
3. 已校准的“R&B、温暖、爱情”请求。
4. 用户选择第二或第三个分支后的延续。
5. 用户只说“不是这个”后的修复。
6. 小结果集、大结果集和零结果三种结果规模。

### 观察指标

- 首轮是否在不必要提问前先提供可用结果；
- Skill 是否明确说出当前理解，且没有把推断写成歌曲事实；
- 分支是否不超过三个、彼此可比较，且没有静默丢掉真实重叠结果；
- 用户纠正时是否只需回答一个真正改变结果的条件；
- 代表性首轮是否保持轻量，并保留明确的继续展开入口；
- 当回复提供继续入口时，是否同时解释“再来一些”和“换一批”的含义，且没有把内部术语变成学习门槛；
- 试听链接、结果状态和匹配证据是否全部来自运行时返回值；
- 实际 token、工具调用次数和完成时间（需要真实运行记录，不能用静态估算代替）。

首轮具体显示几首，以及结果池不足时的降级策略，仍然不在 benchmark 开始前擅自固定；“两种说法都保留当前方向”以及追加/替换差别已经是明确产品规则。

## 下一步决策边界

下一项真正需要产品判断的议题是：当前方向仍有少量有效结果、但不足以完整满足
请求时，应先交付全部剩余结果、只展示代表结果，还是先让用户选择降级路径再
检索。除这项议题外，本轮不再从 review 报告中推导新的交互规则。
