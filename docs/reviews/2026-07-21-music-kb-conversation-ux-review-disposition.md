# Music KB 对话 UX review disposition

日期：2026-07-22（补充记录）

状态：v0.7.5 的对话契约修复和 v0.8.0 的代表性紧凑检索已实施于 Draft PR #46 分支；尚未合并

范围：`plugins/music-kb/skills/music-kb/SKILL.md` 的用户端检索和对话流程，以及为这套流程编写的 conversation-UX metric pack。

不在范围内：周更、下载、CNB、数据库发布、peer 同步、后台脚本复杂度、整包插件的 marketplace 元数据，以及提示词生成功能本身。

## 一句话结论

v0.7.5 修复了分支遗漏、分支平铺和静态指标误标；v0.8.0 进一步把方向发现与候选列表分开，
用后端代表性排序和紧凑投影取代 50-row 完整记录。最终空目录真实复测为 16/16 runtime checks；
每个有证据的方向、后端返回页和最终分组一一对应，跨组重叠与试听链接也满足用户可见契约。
实际 input 比 v0.7.5 样本下降约 62.6%，但 observed-usage 仍为 C / 81、high risk，因此只声称
当前功能验收通过，不声称已建立稳定性能基线。

## 证据边界

### 用户端 Skill（本次 review 的目标）

- `plugin-eval analyze` 目标为 `plugins/music-kb/skills/music-kb`：不带 observed usage 时 A / 100、low risk、0 fail；
  主 Skill invoke 估算 2,178 tokens，deferred follow-up 为 1,431 tokens。
- `conversation-ux` custom metric pack：16/16 静态契约和 16/16 真实行为全部通过，合计 32/32，
  coverage 100%，failed checks 0。
- 加入最终真实样本后为 C / 81、high risk；观测到 146,215 input（其中 123,776 cached）、
  1,681 output、149 reasoning 和 63 秒。这是整个 Codex 宿主会话，不等同于插件载荷。
- 当前 Skill 的静态检查覆盖：分支上限和排序、分支选择后的上下文延续、最小修复问题、渐进式结果展示、追加 / 换批方向和展示语义、用户可见的后续动作指引、证据和 `listen_url` 边界、Markdown 链接与跨组重叠标注、未决语义不得被写成固定默认值，以及自然语言入口示例。
- 真实校准请求和用户确认记录在 [头脑风暴记录](../brainstorms/2026-07-20-music-kb-conversation-ux.md) 的 Round 17；它证明了“温暖”需要至少区分情绪方向和音色方向，但不把这三个方向写成通用分类表。

### 整个插件包（不作为本次 UX 结论）

对 `plugins/music-kb` 根目录运行整包评估得到 26/100，主要提示：

- `plugin.json` 缺少 `websiteURL`、`privacyPolicyURL`、`termsOfServiceURL`；
- 包含所有技能时的静态 deferred/invoke token 估算偏高；
- 后台 Python 文件存在复杂度和长行提示。

这些结果来自插件级目标，不是用户端 retrieval Skill 的行为证据。它们应另开维护任务，不能被写成当前 UX 改动的阻塞项。

## Round 31 追加决策：候选层 → 详情层

用户补充的不是新的检索方向，而是检索完成后的后置详情交互。当前候选结果应保持轻量，
随后用一个简单问题询问“要展开哪些歌曲”；用户可以回复序号、歌名或“前几首”。如果用户
没有指定描述维度，Skill 仍应直接提供完整描述，不要求用户先学习或选择内部字段。

“全部”保留为自然表达：小结果集可以一次展开；较大结果集自动拆成可读的批次，并告诉
用户还可以继续。分批只控制输出体量，不改变当前方向、不重新检索、不静默放宽条件，也不
把未选歌曲的完整分析提前取出。具体触发阈值、每批数量和文本压缩方式留给真实 benchmark，
不在本轮写成固定数字。

## 真实对话 benchmark（第一轮详情体量校准已完成）

详情结果、方法和证据边界见
[完整 benchmark 报告](../benchmarks/2026-07-22-music-kb-conversation-ux-full-analysis.md)。本轮使用
真实只读快照和 canonical analysis；benchmark 当时未修改 Skill 或数据库，当时的
deterministic metric pack 为 8/8 checks。实施后的 pack 已扩展为 12/12 checks、
100% coverage、0 failed checks。

### 目标

验证用户是否能更快得到可用候选、看懂 Skill 采用的理解、低成本纠正方向，并且不会因为库变大而被结果淹没。

### 场景集

1. 清晰的中文标签或别名请求。
2. “有氛围感”这类宽范围主观请求。
3. 已校准的“R&B、温暖、爱情”请求。
4. 用户选择第二或第三个分支后的延续。
5. 用户只说“不是这个”后的修复。
6. 小结果集、大结果集和零结果三种结果规模。
7. 用户选择单首、按序号选择多首、使用“前几首”以及请求“全部”时的详情展开；覆盖小结果
   集一次展开和大结果集自动分批两种体量。

### 观察指标

- 首轮是否在不必要提问前先提供可用结果；
- Skill 是否明确说出当前理解，且没有把推断写成歌曲事实；
- 分支是否不超过三个、彼此可比较，且没有静默丢掉真实重叠结果；
- 用户纠正时是否只需回答一个真正改变结果的条件；
- 代表性首轮是否保持轻量，并保留明确的继续展开入口；
- 当回复提供继续入口时，是否同时解释“再来一些”和“换一批”的含义，且没有把内部术语变成学习门槛；
- 结果不足但仍有有效命中时，是否先交付全部剩余结果，再提供未执行的后续路径；
- 检索后的详情问题是否足够简单，用户能否用序号、歌名、“前几首”或“全部”完成选择；
- “全部”在大结果集上是否自动分批且明确可继续，同时避免一次回复形成长文本结果墙；
- 试听链接、结果状态和匹配证据是否全部来自运行时返回值；
- 实际 token、工具调用次数和完成时间（需要真实运行记录，不能用静态估算代替）。

首轮具体显示几首，以及怎样用统一数字判定“结果不足”，仍然不在 benchmark 开始前擅自固定。结果不足时先交付全部剩余有效结果、再提供两条后续路径，以及“再来一些 / 换一批”的方向保持和追加 / 替换差别，已经是明确产品规则。

## 脑爆收束与后续边界

Round 30 后用户补充了一个重要的后置详情问题；Round 31 只处理其体量边界，现已收束，
不再继续拆分完整描述的字段、排序或文案细节。仍重要但尚未形成默认规则的事项只有：

1. “结果不足”的具体触发阈值；交给真实 benchmark，不预先固定通用数字。
2. 模糊多标签的交集 / 并集语义；保持请求明确时按请求执行，必要时另开产品决策。
3. 首轮具体展示量和预算；等真实运行数据校准，不把当前样例硬编码成数字。
4. 完整描述单批参数已经确定为最多 4 首，并已写入 v0.7.4 Skill。

完整描述的默认语言已经确认，当前没有仍需澄清的阻塞性 UX 项。其余事项继续按上述延后 /
benchmark 边界处理，不再新增 UX 议题；本轮脑爆正式结束。

真实对话 benchmark 是验证任务，不是继续脑爆的前置条件。提示词生成、整包元数据
和后台复杂度属于未来目标或范围外维护，不进入当前 retrieval UX 验收。除非 benchmark
发现阻塞性问题，否则后续直接进入已确认基线的实现 / 验证阶段，不再新增默认规则。

## v0.8.0 代表性紧凑检索与最终验收

- `music_kb_discover` 对完整 canonical 命中集计算 `match_count` 和 facets，不返回歌曲 records；
  `music_kb_recommend` 返回后端代表性排序的紧凑页，并用 `next_offset` 稳定继续。
- 当前真实场景的 discovery + 三个分支载荷为 15,544 bytes，比 v0.7.5 四次完整记录的
  165,201 bytes 下降约 90.6%。后端微基准显示这一取舍只增加约几十毫秒，不是端到端
  63 秒的主要来源。
