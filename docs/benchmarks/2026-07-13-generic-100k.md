# Synthetic generic-import benchmark: 100k records

This is a reproducible **synthetic** benchmark for the local SQLite database.
It contains no production Music Flamingo analysis, lyrics, audio, or KuGou
delivery data.

## Command

From `plugins/music-kb`:

```bash
uv run python scripts/benchmark_100k.py --records 100000 --repetitions 20
```

The script writes a physical-LF JSONL file, streams it through the generic
importer with `--batch-size 500`, performs its one final FTS5 rebuild, checks
validation, and measures warm local reads. The default temporary workspace is
removed afterward.

## Workload

- 100,000 canonical synthetic recordings;
- one title identity tag and one artist identity tag per recording;
- four reusable analysis tags per recording (`electronic pop`, `sidechain`, a
  bounded synth-texture family, and a bounded mood family);
- one rare exact tag every 997 records;
- one numeric BPM feature per recording;
- 20 warm query repetitions, returning at most 10 records.

## Local result (2026-07-13)

Host/runtime: macOS Darwin 25.2.0 arm64, Python 3.12.13, SQLite 3.50.4.

| Measure | Result |
| --- | ---: |
| JSONL generation | 0.487 s |
| Generic import plus one FTS rebuild | 27.721 s |
| Input JSONL size | 77,511,585 bytes |
| Finished main SQLite size | 297,271,296 bytes |
| Rare exact tag median / p95 | 0.810 / 0.928 ms |
| Common exact tag median / p95 | 270.851 / 342.361 ms |
| Full-text median / p95 | 228.203 / 262.191 ms |
| Exact title median / p95 | 30.386 / 30.949 ms |
| Artist median / p95 | 60.437 / 61.687 ms |

The common-tag result intentionally has 100,000 candidate records, so it
measures broad-result ordering rather than a rare-tag lookup. The important
retrieval guarantee for this KB is that granular/rare exact tags remain
sub-millisecond at 100k; broad local queries remain below one second on this
machine.

These are local engineering measurements, not a hardware-independent SLA.
Run the command again after significant schema, SQLite, or hardware changes.
