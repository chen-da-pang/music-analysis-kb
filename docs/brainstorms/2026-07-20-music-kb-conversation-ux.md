# Music KB 对话 UX 头脑风暴记录

状态：当前设计切片已批准，进入实现（未决语义继续保持未决）
开始日期：2026-07-20
关联 Issue：[Issue #41](https://github.com/chen-da-pang/music-analysis-kb/issues/41)

## 这份文档记录什么

这是一份追加式的设计讨论记录。每一轮只记录已经发生的事实、用户做出的决定、支持决定的证据和仍未解决的问题；不把未讨论的方案写成既定需求。

本记录的对象是 `music-kb` retrieval Skill 的用户对话流程，不是数据库、标签抽取、周更、CNB、Feigua 或生成提示词流程。

## 当前问题

用户可以用自然语言说“找几首带某个标签的歌”，但现有 Skill 主要说明内部检索工具和调用顺序，没有明确规定用户看到的交互流程。

因此，第一次使用的用户可能需要猜测：

- 是否必须知道内部的英文 canonical tag；
- “一些/几首”应如何理解；
- 多个标签是交集还是并集；
- 没有精确命中时，Skill 会不会换一种理解；
- 零结果代表没有数据、标签不存在，还是查询条件太窄；
- 用户纠正一次理解后，Skill 是否会直接修正并继续。

核心 UX 问题不是“教用户记住更多命令”，而是让用户在不学习内部实现的情况下完成一次自然、可控、可恢复的检索。

## 已有证据

### Issue #38 / Draft PR #39

Issue #38 正确指出了自然语言请求的学习门槛，但随后的 Draft PR #39 很快把若干解决方案写成了固定规则，包括默认数量、标签回退顺序和澄清边界。

PR #39 的结构和静态规范检查结果良好，但用户随后明确不同意这次直接实施，并要求撤回、重新讨论。这说明“规则完整”不等于“交互流程已经被用户认可”。

Issue #38 与 PR #39 均保留为历史讨论，不代表已采用的产品决策。

### 当前讨论得到的 UX 方向

用户明确表示：这是一项对话 UX 优化，而不是传统 UI 优化，也不是制作一套教用户学习插件的教程。

## 头脑风暴轮次

### Round 0：问题重新定义

**触发：** 发现用户可以提出标签歌曲请求，但 Skill 没有用户侧交互规范。

**讨论结论：** 问题属于“自然语言检索的对话流程设计”，重点是学习门槛、澄清、反馈、证据和恢复，不是 MCP 工具说明的补充。

**被否定的做法：** 在没有先讨论交互方案的情况下直接修改 Skill、默认提示和测试。

**保留边界：** 不改变 retrieval-only、read-only、Music Flamingo-only、无 Feigua、无 Suno prompt conversion 的既有边界。

### Round 1：能力盘点与范围收窄

**讨论过的能力：**

- `Product Design Research`：从 Issue、PR、真实问答和反馈中找出实际摩擦；
- `Plugin Eval: metric-pack-designer`：把已确认的 UX 原则变成可重复检查的指标；
- `cognitive-adaptation`：帮助降低回答理解成本；
- `brainstorming`：保证先讨论、再确认、后实施；
- `visualize`：在 Codex 对话中承载可点击的方案选择；
- 外部的 Model Interaction Design、UX Writing、Chatbot Flow Design。

**用户纠正：** Model Interaction Design 中的 progressive disclosure 等内容更像教用户发现和学习 AI 能力，不是本任务首先要解决的交互流程问题，因此不作为核心依赖。

**当前保留的组合：**

1. `Product Design Research` 负责回答“用户实际在哪里卡住、为什么卡住”；
2. `Plugin Eval: metric-pack-designer` 负责回答“确定方案后，怎样判断它真的变好了”。

这两个能力目前只作为讨论和后续评估工具，尚未安装、扩展或执行。

### Round 2：第一项交互决策

关联 Issue：[Issue #41](https://github.com/chen-da-pang/music-analysis-kb/issues/41)

**用户选择：A**

当用户说“找几首带某个标签的歌”时，Skill 默认先按当前理解检索，并在结果中说明自己的理解；只有歧义会实质改变结果时，才先提一个澄清问题。

**这项决定解决的方向：**

- 用户不必先学习 canonical tag、MCP 工具名或 CLI 参数；
- 清晰请求不被不必要的确认步骤打断；
- Skill 仍然要对真正影响结果的歧义交还控制权；
- 结果中的“我的理解”必须可被用户快速纠正。

**尚未决定的内容：**

- 什么程度算“足够清晰”；
- 结果中如何展示精确命中、别名命中和近似匹配；
- 默认返回多少首；
- 多标签的交集/并集表达方式；
- 零结果、快照不可用和试听链接缺失时的具体回复；
- 用户纠正后是否自动重试，以及重试到什么边界。

这些内容必须在后续轮次逐项讨论，不能因为 A 已确定就自动推导成固定实现规则。

### Round 3：澄清触发阈值

**用户选择：3（按影响判断）**

在 A 模式下，只要存在一个合理的主要理解，就先按这个理解检索；只有存在多个合理理解，且这些理解会实质改变返回结果时，才先向用户澄清。

**这项决定进一步明确了 A：**

- “有一点模糊”本身不是提问理由；
- 提问的理由是不同理解会导致明显不同的结果范围、筛选条件或用户行动；
- 能够先给出有用候选时，不用把内部分类问题转嫁给用户；
- 结果中仍要让用户看见 Skill 采用了哪一种理解，从而可以快速纠正。

**尚未决定的内容：**

- 什么叫“实质改变结果”；
- 主观描述（如“有氛围感”“适合夜晚开车”）是否通常属于可先检索的请求；
- 多个标签之间的含义差异何时足以触发澄清；
- 澄清问题是否需要同时给出当前理解和可选方向。

### Round 4：头脑风暴推进方式（过程决定）

**用户纠正：** 每轮用户做出选择后，助手应在同一轮完成记录、同步 Issue，并立即提出下一轮唯一议题；不能停下来等待用户再次说“继续”。

**采用的过程规则：** 后续每轮都按“记录本轮决定 → 更新 Issue → 提出下一题”的连续节奏推进。下一题仍然只问一个问题，等待用户回答后再记录下一轮。

### Round 5：主观宽范围请求的处理方式

**用户决定：先分支检索，再在回答中澄清。**

面对“找一些有氛围感的歌”这类可能对应多个合理方向的请求，Skill 不在检索前拦住用户。它先对几个合理方向分别检索，用真实结果让用户看见不同理解，再在同一条回答中请用户选择或细化需求。

**采用的交互原则：**

- 先提供实际价值，再要求用户补充信息；
- 澄清不再是一道开始工作前的门槛，而是基于初步结果的细化；
- 每个分支必须明确说明代表哪一种理解，不能把不同方向混成一个列表；
- 用户选择方向后，再沿所选方向继续检索，而不是重新从零询问需求。

**尚未决定的内容：**

- 首轮分支应该只给轻量样本，还是完成用户要求的全部数量；
- 默认展示几个分支；
- 如何选择最有区分度的分支；
- 用户没有继续回复时，首轮结果是否已经足够独立使用。

### Round 6：首轮分支的完成深度

**用户选择：2（每个分支都完成用户要求的数量）。**

当请求存在多个合理理解时，首轮不只展示试探样本，而是让每个被选中的方向都完成用户原本要求的结果量。用户可以直接比较完整的方向结果，再决定后续要沿哪一个方向细化。

**这项决定带来的约束：**

- 分支结果必须分别标明它们代表的理解；
- 不能用“完整”掩盖分支之间的差异，也不能把多个方向混成一个总列表；
- 首轮回答需要控制结构和重复信息，避免分支数量失控；
- “完成用户要求的数量”不等于已经确定了“几首”的具体默认值，数量规则仍待讨论。

**尚未决定的内容：**

- 合理方向超过两个时最多展示几个完整分支；
- 分支如何排序，是否需要一个默认主方向；
- 多个分支的结果是否需要去重；
- 用户不再回复时，多个完整分支是否都作为最终可用结果保留。

### Round 7：首轮分支数量上限

**用户选择：2（最多 3 个完整分支）。**

当合理理解超过两个时，首轮最多展示三个完整分支。超过上限的方向不在首轮无限扩张，而应依据区分度和相关性筛选；未展示的方向不能被默认为不存在。

**这项决定的目的：**

- 保留“先把活做完”的价值；
- 防止合理解释数量增长后，回答变成无法比较的结果墙；
- 让后续的排序、去重和用户选择仍有清晰边界。

**尚未决定的内容：**

- 三个分支按什么顺序呈现；
- 是否明确标记“最可能理解”；
- 未进入前三的合理方向如何告知用户；
- 分支之间出现同一首歌时如何处理。

### Round 8：分支呈现顺序

**用户选择：1（按最可能的理解排序）。**

首轮分支按当前判断的相关性从高到低呈现：最可能的理解放在第一位，其他合理方向作为备选。这个排序是为了让用户最快拿到最可能有用的结果，不表示第一项是客观唯一答案。

**这项决定的约束：**

- 第一项必须明确标注为“当前最可能的理解”或等价表达；
- 备选方向仍需完整呈现，不能因为排序靠后就省略；
- 排序依据应可解释，不能把模型偏好伪装成数据事实；
- 用户选择其他方向后，应沿该方向继续，而不是要求用户重新描述原请求。

**尚未决定的内容：**

- 是否展示排序依据或置信度；
- 每个分支的理解说明需要多长；
- “最可能”是否需要用户主动确认；
- 分支之间重复歌曲的处理方式。

### Round 9：最可能理解的透明说明

**用户选择：2（标注理解并给出简短依据）。**

当第一项分支是当前判断中最可能的理解时，回答要明确说明采用了什么理解，并用一句简短依据解释为什么排在第一位。不使用未经验证的数字置信度，也不把模型判断写成客观事实。

**这项决定的约束：**

- 说明要服务于用户纠正和选择，不变成冗长推理；
- 依据必须来自用户请求或可见检索条件，不能编造用户意图；
- 备选分支也要有清晰的理解标签，但不必重复同样长度的解释；
- 用户否定第一项理解后，直接沿其选定方向继续，不要求重新描述整句请求。

**尚未决定的内容：**

- 同一首歌同时符合多个分支时如何展示；
- 依据是否需要在每个分支都出现；
- 用户只说“不是这个”时的最小修复动作。

### Round 10：重叠结果的展示

**用户选择：1（在每个分支都保留，并明确标注它也符合其他方向）。**

当同一首歌同时符合多个分支时，在每个相关分支中保留这首歌，并在条目旁明确标注它也符合哪些其他方向。不静默去重，也不让重复出现看起来像检索错误；每个分支仍然保持独立可用，用户可以直接比较各方向的完整结果。

**这项决定的约束：**

- 交叉标注要简短、可扫描，不重复粘贴其他分支的整段解释；
- 不能因为跨分支去重而破坏“每个分支完成用户要求数量”的决定；
- 分支内部的排序和结果证据仍应保持可理解，重叠标注只是补充关系说明；
- “重复歌曲是否计入每个分支的数量”“同一分支内是否还需要单独的去重展示”仍未决定。

### Round 11：选定分支后的轻量延续

**用户选择：1（轻量改写版）。**

用户选定某个分支后，默认把该分支设为当前对话上下文，保留现有结果，不自动重新检索。只有用户明确提出新的检索或分析动作（例如再找一些、换一批、收窄条件、扩展方向或深入分析）时，才继续处理。

**这项决定的约束：**

- “选择分支”表示选择当前方向，不等于请求一次新的检索；
- 确认当前方向时不再追加“保留 / 扩展 / 分析”的第二层菜单；
- 后续回合可以复用已选分支，用户不必重复描述原始需求；
- “已交付”不作为用户侧的结束语，避免让对话显得被关闭；
- 何种具体措辞算作新的检索或分析动作，仍待后续精炼。

### Round 12：用户否定后的最小修复

**用户选择：2（只问一个最小澄清问题，再继续）。**

当用户只说“不是这个”时，Skill 不直接猜测另一个分支，也不要求用户重新描述完整需求；它只针对一个会改变结果的关键轴提问，再按用户的回答继续。这样既承认当前理解不对，也不把修复成本全部推回给用户。

**这项决定的约束：**

- 澄清问题必须基于当前请求和已经展示的结果，不能重新开启一轮完整需求采集；
- 一次只问一个问题，优先询问最能区分候选方向的条件；
- 等待回答期间保留已有结果和上下文，不先盲目扩展新的分支；
- 不要求用户学习 canonical tag 或内部检索术语；
- 当无法找到一个足以区分结果的最小问题时，后续如何降级仍待决定。

### Round 13：首轮结果量的可扩展处理

**用户选择：1+（固定轻量首轮 + 按结果集分层展开）。**

数据库整体规模变大，不应自动让第一次回答变长。方案 1+ 将“首轮负担”和“后续可达性”分开：命中结果少时直接展示全部；命中结果多时只展示固定少量、覆盖不同子方向的代表性候选，并让用户通过“再来一些 / 换一批”等明确动作继续追加；用户明确要求“全部”或“尽可能多”时，再切换为分组、分批的完整结果模式。

**这项决定的约束：**

- 当前约 1400 首、单周约新增 400 首，不改变首轮交互的基本形状；
- “不漏选项”分别由代表性覆盖和可继续展开来解决，不靠一次性输出全部歌曲；
- 小结果集不应被固定上限截断，大结果集也不应因为库增长而无限膨胀；
- 继续追加时沿用当前分支、排除已展示歌曲，不重新澄清原请求；
- 固定首轮的具体数量、代表性候选的选择标准、完整结果模式的具体分组方式仍待决定。

### Round 14：首轮数量的校准方式

**用户选择：3（先用真实请求样例校准，不立即锁定具体数字）。**

方案 1+ 的“少量首轮”先作为交互原则保留，但暂不把 3 首或 5 首写成固定规则。先用不同类型的真实请求观察首轮是否足够有用、是否遗漏明显方向，以及用户是否自然地提出“再来一些”，再决定具体数量。

**这项决定的约束：**

- 数量暂缓不等于改用完全动态、不可预测的返回量；首轮仍须保持轻量上限；
- 校准样例应覆盖清晰标签、主观宽范围请求、多个分支和小结果集等情况；
- 观察重点是用户能否理解和继续操作，不只看一次回答列出了多少首歌；
- 在数量确定前，不把示例中的 3 或 5 首写成 Skill 验收规则。

### Round 15：代表性候选的选择原则

**用户选择：3（相关性为主，同时保留最低限度的方向覆盖）。**

首轮候选先满足用户当前描述的相关性，再在不牺牲明显相关性的前提下，至少保留一个不同的子方向。多样性是防止结果过于同质的最低保障，不取代“最可能有用”这一排序原则。

**这项决定的约束：**

- 不能为了凑覆盖而加入明显不相关的歌曲；
- 如果结果池太小，或没有可信的不同子方向，就不强行制造多样性；
- 子方向标签必须来自可见检索依据，不能把模型猜测写成事实；
- 具体如何向用户说明“代表性”以及如何选择校准样例，仍待决定。

### Round 16：代表性结果的透明说明

**用户选择：1（在分支开头用一句话说明，并提示可以继续展开）。**

当首轮结果只是代表性候选而不是完整列表时，在分支标题或结果开头加一句简短说明，让用户知道当前回答先提供一小组可用候选，并且可以沿当前方向继续。说明不展开内部排序逻辑，也不新增一层操作菜单。

**这项决定的约束：**

- 说明要短且直接，不能让用户误以为必须学习“代表性”规则；
- 小结果集已经全部展示时，不应错误地称为“代表性候选”；
- 提示继续展开是一个轻量入口，不要求用户先选择多个后续动作；
- 具体“再来一些”和“换一批”的后续行为仍待决定。

### Round 17：真实检索校准与实施授权

**真实请求：**“我需要一些 R&B、温暖的、关于爱情的歌。”

**只读检索证据：** 当前快照 `music-kb-2026w29-kugou-431-final`
包含 1,348 条 recording / canonical analysis，检索投影状态为 `current`。
使用 `r&b + warm + love` 检索时至少返回 50 首并触及单次调用上限；
前 50 首中有 45 首带 `melancholic`、33 首带 `ballad`、30 首带
`romantic`、49 首带 `intimate`，但只有 7 首带 `soul`。这说明 `warm`
在当前库中经常描述音色或制作质感，不能直接等同于“爱情情绪也温暖”。

**用户确认的三方向：**

1. **情绪也温暖**：偏安心、陪伴、希望感；真实细化检索
   `r&b + warm + love + romantic + hopeful` 命中 12 首。
2. **音色温暖，但爱情内容可以感伤**：偏思念、失落、忧郁；真实细化检索
   `r&b + warm + love + romantic + melancholic` 命中 27 首。
3. **仍在请求范围内、但更跳脱的 Soul / Neo-soul 质感**：真实细化检索
   `r&b + warm + love + soul` 命中 7 首。

三方向既解决“音色温暖 / 情绪温暖”的歧义，也保留一个可信且有区分度的
不同子方向。它们不是固定写死的通用标签组合，而是本次真实请求验证出的分支
示例；其他请求仍要依据用户原话和实际检索证据生成最多三个合理分支。

**用户批准：** 用户确认这三个方向符合预期，并明确授权按照本轮和前 16 轮
已确认的内容开始实施。

**本次实施边界：**

- 将已确认的检索时机、分支、排序、交叉标注、轻量首轮、上下文延续和最小修复
  规则写入 retrieval Skill、插件入口文案和回归评估；
- 用确定性 custom metric pack 检查这些已确认的 UX 原则；
- 继续保持 read-only、retrieval-only、Music Flamingo-only，并保留真实
  `listen_url`；
- 不在本次实现中自行决定“再来一些 / 换一批”的追加或替换语义、多个标签的
  交并集默认含义、未校准的固定首轮数量，以及其他仍未讨论的行为。

### Round 18：后续请求始终保留当前方向

**用户决定：** “再来一些”和“换一批”都保留用户已经选定的方向。两种说法
都不新增一个理解方向、不自动切换到其他分支，也不要求用户重新描述原请求。

**术语修正：** “追加分支”是不准确的说法。这里的“分支”指一种请求理解，
例如“情绪温暖”或“音色温暖但可以感伤”。本轮真正要讨论的是“在当前分支内
继续给新结果”，而不是再建立一个新分支。

**已经确定的行为：**

- 后续检索沿用当前方向和已有条件；
- 新结果应尽量排除已经展示过的歌曲；
- “再来一些”和“换一批”的差别不能建立在是否换方向上。

**仍未确定：** 两个动作如何处理旧结果的展示。尤其是“再来一些”是否保留旧
结果并在后面增加新结果，以及“换一批”是否用新结果替换当前展示、同时仍保留
方向和对话历史。这一项需要单独确认后才能进入 Skill。

### Round 19：追加与换批的展示差别

**用户选择：方案 1。**

- **再来一些：** 沿用当前方向，保留已经展示的结果，并在后面追加新的、尚未
  展示过的歌曲。
- **换一批：** 沿用当前方向，用新的一批、尚未展示过的歌曲替换当前展示；方向
  和对话上下文继续保留，旧结果仍属于此前的对话记录。

这里的“追加”指增加当前方向的歌曲结果，不是增加一个新的理解分支。两种动作
都不切换方向、不静默放宽条件，也不要求用户重复原始请求。

**本轮实现边界：** 只把上述追加 / 替换语义写入 retrieval Skill、入口文案和
回归 metric；结果池耗尽时的进一步降级策略、多标签交集/并集默认语义和首轮具体
数字仍然不在本轮决定。

### Round 20：把后续动作语义直接告诉用户

**用户决定：** 如果 Skill 规定了“再来一些”和“换一批”的语义，实际回复在
提供可继续展开的结果时，也必须同时给用户一个简短、白话的使用指引。用户不
应该为了发现这两个说法的含义而阅读 Skill 或猜测内部规则。

**用户可见指引：**

> 你可以这样继续：
> - “再来一些”：保持这个方向，保留已展示的歌，再补充一批之前没展示过的歌。
> - “换一批”：保持这个方向，换一批之前没展示过的新歌，替换当前展示；之前的结果仍留在对话记录里。

**这项决定的约束：**

- 指引紧跟在“代表性结果 / 当前方向可以继续展开”的说明之后，不藏在内部
  规则中，也不要求用户学习 branch、cursor 或检索限制等术语；
- 用户实际使用其中一个说法时，如果行为可能不明显，可以用一句普通话确认
  正在追加还是替换，但不静默切换方向；
- 无关回复或没有提供继续入口时不重复完整教程，避免把轻量检索变成操作手册；
- 指引只解释已经决定的追加 / 替换语义，不提前决定结果池耗尽时的降级方式。

**本轮实现边界：** 将这段用户可见指引加入 retrieval Skill 的 response
contract、插件 README 和 conversation-UX 回归 metric，并升版到 0.7.3；不
改变检索、排序、结果池或数据库行为。

**下一轮讨论：** 当前方向没有足够未展示结果时，Skill 应该如何处理：
1. 明确告知当前方向没有更多未展示结果并停止；
2. 告知结果不足，再询问用户是否愿意放宽一个条件；
3. 提供相邻方向候选，但明确说明已经改变了条件。

### Round 21：结果不足时保留“放宽条件”和“相邻方向”两条路径

**用户选择：2 和 3。**

当当前方向没有足够的未展示结果时，Skill 不应静默放宽条件，也不应自动切换
到相邻方向。它应先明确告诉用户当前方向的结果已经不足，并保留两种由用户
决定的后续路径：

1. **放宽一个条件：** 先说明准备放宽哪一个条件，等待用户确认；
2. **探索相邻方向：** 提供相邻方向候选，并明确指出相对于当前方向改变了什么。

这次选择只确定“两条路径都要保留”，还没有决定它们是按先后出现、同屏出现，
还是先取一小组相邻方向样例再让用户决定；也没有把“候选”定义成仅显示方向
名称，还是直接展示真实歌曲。后续检索不得在用户选择前自动改变条件。

**本轮实现边界：** 先记录组合决策，不修改 Skill 的结果不足处理；下一轮只
讨论两条路径的呈现顺序、是否同屏，以及相邻方向是否需要先做轻量取样。

## 当前设计假设（待验证）

1. 用户的主要目标是尽快得到可用的歌曲候选，而不是学习 Music KB 的分类系统。
2. 对清晰请求，立即检索比先询问内部标签更符合用户预期。
3. 透明说明“我如何理解你的请求”比默默扩大查询范围更容易让用户纠正。
4. 研究证据和评估指标必须分开：前者描述问题，后者衡量方案；两者都不能替代用户对交互方向的确认。

### Round 22：并列给出两条后续路径，但先不检索新方向

**用户选择：2（并列呈现）。**

结果不足时，同一条回复应先说明当前方向为什么无法继续提供足够的新结果，
然后简洁列出：

- 一个基于当前请求、可以明确放宽的条件；
- 一至两个相邻方向，并说明它们相对当前方向改变了什么。

这些内容是供用户选择的路径，不是已经执行的检索结果。用户选择前不检索新方向、
不放宽条件，也不替用户选择；用户选择后才沿对应路径继续。回复仍应保留当前
方向和已有对话上下文，不能把并列选项写成新的理解分支或暗示用户必须重述原请求。

**仍未确定：** 如何选择“一个可放宽条件”和“一至两个相邻方向”，以及相邻
方向候选是否需要统一的数量上限和命名规则。

**本轮实现边界：** 先记录并列呈现决策，不修改 Skill；下一轮只讨论并列选项的
生成边界和数量，避免把结果不足提示变成新的结果墙。

### Round 23：并列选项数量采用完全动态策略

**用户选择：3（完全动态）。**

并列选项的数量不预先固定为“两项”或“三项”，而是根据当前请求、已有检索
证据和真正有区分度的相邻方向动态决定。没有足够证据的方向不应为了填满选项
而出现；同样，动态数量也不代表可以无上限地堆出结果墙。

**这项决定的约束：**

- 至少保留一个明确可理解的“放宽条件”路径；
- 相邻方向只有在当前证据支持、且与当前方向有实际区别时才出现；
- 每个选项都要说明改变了什么，不能只列内部标签或抽象分类名；
- 用户选择前仍不检索、不放宽、不自动切换；
- 仍需下一轮确定动态策略的停止护栏：内容预算、证据阈值，或分层展开方式。

**本轮实现边界：** 只记录动态数量决策，不把它写入 Skill；下一轮专门决定
如何让“动态”保持轻量、可预测且不会变成结果墙。

### Round 24：优先一次回答完整覆盖，不把展开动作变成必要步骤

**用户决定：** 回答应尽可能简单，同时覆盖有用且彼此不同的内容。只要能够在
一次回答中清楚、轻量地把问题回答好，就不应为了所谓“分层展开”故意扣住内容，
再要求用户主动说“展开”才能看到。

**采用的停止原则：**

- 优先压缩每个选项的表达，而不是优先减少选项数量；
- 首轮一次给出所有有证据、对用户确实有不同价值、且能保持可扫描的路径；
- 后续路径已经重复、证据弱，或继续加入会明显破坏可读性时才停止；
- “还有更多”只能用于次要、重复或低优先级内容，不能隐藏本可以在当前回答中
  简洁交付的重要方向；
- 仍保持 Round 22 的边界：这些是供用户选择的路径，选择前不执行新的方向检索。

这相当于用“证据 + 差异 + 可扫描性”共同约束动态数量，而不是采用固定数量，
也不是默认采用分层展开。目标是一轮足够完整，但不把回答写成结果墙。

**本轮实现边界：** 先记录一次回答优先原则，不修改 Skill；下一轮讨论判断两个
方向是否真正不同的标准。

### Round 25：方向必须同时具备证据差异和用户选择价值

**用户选择：3。**

两个候选方向只有同时满足以下条件，才值得在同一回答中分别出现：

1. 底层检索结果、标签共现或可验证分析确实支持它们存在差异；
2. 这种差异会改变用户听到的内容、感受到的氛围，或最终会选择哪一组结果。

只有标签不同、但用户实际得到的体验相同，应合并成一个方向，并把底层差异压缩
为该方向内的补充说明。只有用户侧描述听起来不同、但没有检索证据支撑，也不能
单独制造一个方向。方向名称必须使用用户可理解的差异，而不是只展示内部标签。

这条双重门槛与 Round 24 的“一次回答完整覆盖”配合：重要且真正不同的方向
首轮一次给全；技术上不同但用户价值重复的内容不占用独立选项。

**本轮实现边界：** 先记录方向差异判定标准，不修改 Skill；下一轮讨论结果不足
时应优先建议放宽哪一个条件。

### Round 26：建议放宽条件时优先保护用户意图

**用户选择：2（用户意图优先）。**

结果不足时，Skill 应优先建议放宽用户原请求中最不核心的条件，而不是选择能
带来最大结果增量的条件。用户已经明确强调、重复说明、选定为当前方向，或在
纠正中专门保留的条件，不能仅因为放宽后结果更多就被优先移除。

建议必须用用户可理解的语言说明“准备放宽什么、哪些核心条件继续保留”。结果
数量是判断建议是否有用的参考，但不能凌驾于用户当前意图之上。若无法判断哪个
条件更次要，不能把模型猜测伪装成用户偏好。

**本轮实现边界：** 先记录意图优先原则，不修改 Skill；下一轮讨论判断条件核心
程度时应使用当前句子、完整对话，还是默认向用户提问。

### Round 27：使用完整对话判断条件核心程度

**用户选择：2（完整对话优先，仍不明确时最小提问）。**

Skill 判断哪个条件最不核心时，不能只读取用户当前一句，也不能在每次结果不足时
把判断工作全部交还给用户。它应使用完整对话中的已知选择和修正，按以下顺序判断：

1. 当前已经选定的方向，以及用户纠正后明确要求保留的条件；
2. 用户明确强调或重复表达的条件；
3. 用户原始措辞中可以直接确认的偏好。

后来的明确纠正可以更新早先的理解，不能把互相冲突的旧表述和新表述机械叠加。
“再来一些”“换一批”等后续说法也继续继承当前方向，不因为当前一句缺少标签就
丢失此前上下文。只有使用完整对话后仍无法区分哪些条件更核心，才问一个真正会
改变放宽选择的最小问题；用户回答前不放宽条件、不切换方向，也不把推测写成
用户偏好。

**本轮实现边界：** 只记录核心程度的判断来源和优先级，不修改 Skill；下一轮
讨论无法区分时，这一个最小问题应直接让用户选择、先给建议再确认，还是并列
说明两个候选条件后让用户选择。

### Round 28：无法判断时并列两个放宽条件

**用户选择：3（并列两个候选条件）。**

只有在完整对话仍无法判断哪个条件更不核心时，Skill 才提出这个最小问题。它应
并列给出两个可放宽的候选条件，各用一句用户可理解的话说明：放宽什么、其他哪些
条件继续保留，以及这会怎样改变结果，然后让用户选择。

**这项决定的约束：**

- 两个候选保持中性并列，不先替用户推荐或暗示其中一个更符合其偏好；
- 解释的是用户会感受到的结果变化，不只展示内部标签或预估结果数量；
- 仍然只问这一个问题，不在同一回合追加其他澄清；
- 用户选择前不执行放宽后的检索、不切换方向，也不把任一候选写成已确认偏好；
- 用户的选择就是对本次放宽条件的确认，不需要再加一轮重复确认。

**本轮实现边界：** 只记录最小问题的呈现方式，不修改 Skill；下一轮讨论当前方向
仍有少量有效结果、但不足以完整满足请求时，是否应先把这些真实结果交付给用户。

### Round 29：结果不足时先交付全部剩余有效结果

**用户选择：1（先交付全部剩余有效结果）。**

当前方向仍有少量尚未展示的有效结果、但已经不足以完整满足请求时，Skill 应先
简短说明当前结果有限，然后把这些仍然符合当前方向的歌曲全部交付给用户。不能
因为结果数量不理想就扣住真实命中，也不能为了凑足数量静默放宽条件。

交付完剩余结果后，再简洁列出已经确定的两条后续路径：放宽一个条件，或探索
相邻方向。两条路径仍然只是供用户选择的下一步，用户选择前不执行新的检索、不
混入部分符合的新歌，也不重复已经展示过的当前方向结果。

**这项决定的约束：**

- “全部”只指当前方向中尚未展示、且证据仍然支持的有效结果；
- 匹配依据、试听链接和结果状态继续来自真实运行时返回值；
- 结果不足的说明保持简短，不把部分成功写成失败，也不隐藏数量有限的事实；
- 本轮不定义统一的“结果不足”数字门槛，也不改变小结果集全量、大结果集代表性
  首轮的既有规则。

**本轮实现边界：** 只记录部分结果的交付顺序，不修改 Skill；下一轮讨论用户选择
放宽条件或相邻方向后，该选择是否成为后续“再来一些 / 换一批”的当前方向。

## Round 30 前提出的议题（已完成选择）

用户选择“放宽一个条件”或某个“相邻方向”后，后续对话应怎样保存这个选择？

1. **成为新的当前方向：** 立即按用户选择检索，后续“再来一些 / 换一批”都沿用
   新方向，直到用户再次纠正或切换。
2. **只对下一次检索生效：** 本次按选择执行，之后自动回到原来的严格方向。
3. **同时保留新旧两个方向：** 后续每次都让用户说明要继续哪一个方向。

多标签交集 / 并集语义和首轮具体数字继续不写成默认规则。

### Round 30：用户选择后的方向成为当前方向，并结束本轮脑爆

**用户选择：1。**

用户选择“放宽一个条件”或某个“相邻方向”后，应立即按该选择继续检索，并把它
设为新的当前方向。之后的“再来一些 / 换一批”沿用这个新方向，直到用户再次
纠正或主动切换。原来的严格方向不被删除，仍保留在对话历史中，但不再作为默认
的活动方向。

这个选择本身就是对后续路径的确认，不需要再增加一轮确认；也不意味着 Skill
可以忘掉原始请求或静默扩大新的方向。

**用户补充的流程决定：** 当前脑爆已经足够深入。完成本轮记录和未决项审计后，
不再继续提出新的 UX 议题；剩余事项明确标记为 benchmark、独立产品决策或范围
外工作，不在本轮凭空补规则。

## 脑爆收束（本轮结束）

### 已形成的可实施 UX 基线

当前基线已经覆盖：先检索后澄清、宽泛请求的真实分支、分支选择后的上下文、最小
纠正问题、轻量而完整的首轮、 “再来一些 / 换一批”的追加与替换、结果不足时的
两条用户选择路径、用户意图优先、完整对话判断、并列澄清、部分结果先交付，以及
选择后新方向的持续继承。

### 仍重要但不在本轮继续澄清的事项

1. **“结果不足”的具体触发阈值：** 不固定一个通用数字，交给真实 benchmark
   观察不同请求和库规模下的可用性，再决定是否需要产品规则。
2. **模糊多标签的交集 / 并集默认语义：** 当前只按足够明确的自然语言请求执行，
   不把某一种逻辑强行写成通用默认；需要时另开产品决策。
3. **首轮具体展示量和预算：** 已确定小结果集全量、大结果集代表性展示，但不在
   没有真实运行数据前固定具体首轮数字。
4. **真实对话 benchmark：** 用于验证自然度、token、调用次数、完成时间和结果
   墙风险，是验证工作，不再作为本轮脑爆问题继续拆解。

提示词生成、插件整包元数据 / token 估算、后台脚本复杂度、周更、数据库、CNB、
peer 和音频流程均保持现有范围边界，不属于本轮用户端 retrieval UX 的未决项。

**收束规则：** 以上未决产品 / 验证事项不得被模型自行猜测成默认行为；除非真实
benchmark 暴露出阻塞问题，否则本轮脑爆到此结束，后续进入已确认设计的实现或
验证阶段。

## 记录规则

- 每轮讨论追加一个 `Round N`，不覆盖历史决定。
- 用户明确否定的方案保留在历史中，并标注为否定，不删除上下文。
- “建议”“假设”“待验证”不能写成“已决定”。
- 只有用户确认的交互原则，才可进入后续设计文档或实现 Issue 的验收标准。
